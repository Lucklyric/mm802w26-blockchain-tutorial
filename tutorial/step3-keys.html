Step 3: ECDSA Key Pairs

Public-key cryptography lets you prove your identity without sharing a secret. You generate a private key (kept secret) and a public key (shared with everyone).

How It Works

  • Private key (SigningKey) — A 32-byte secret number. Used to sign data.
  • Public key (VerifyingKey) — Derived from the private key. Used to verify signatures.
  • ECDSA — Elliptic Curve Digital Signature Algorithm
  • secp256k1 — The specific curve used by Bitcoin (and this tutorial)

Your Key Generation Process

  email + student_random + instructor_random
                      |
                      v
              +---------------+
              |   SHA256       |
              |   (32 bytes)   |
              +-------+-------+
                      | seed
                      v
              +---------------+
              |  SigningKey    |---- Private Key (keep secret!)
              |  (secp256k1)  |
              +-------+-------+
                      |
                      v
              +---------------+
              | VerifyingKey  |---- Public Key (share freely)
              +---------------+

Exercise: Part 1

Open client/part1_keygen.py and implement two functions:

generate_seed()

  1. Concatenate: EMAIL + str(STUDENT_RANDOM) + str(INSTRUCTOR_RANDOM)
  2. Hash with SHA256
  3. Return the raw bytes (.digest(), not .hexdigest())

generate_keypair(seed_bytes)

  1. Create a SigningKey from the seed bytes using SECP256k1
  2. Get the VerifyingKey from the signing key
  3. Return both as a tuple
Hint: The seed is exactly 32 bytes (SHA256 output), which is the right size for a secp256k1 private key.

Verification

cd client
uv run python part1_keygen.py

You should see:

  • A seed hex string (64 characters)
  • A public key hex string (128 characters)
  • A .keys file created in the client directory
Keep your .keys file safe! If you delete it, you'll need to re-run Part 1. The keys are deterministic (same email + randoms = same keys), so you can always regenerate them.
